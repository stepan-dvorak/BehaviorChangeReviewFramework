# Purpose

This note evaluates **ISO/IEC/IEEE 42030:2019 — Software, systems and enterprise — Architecture evaluation framework** as the closest existing standard to the proposed Behavior Change Review Framework.

The central questions are:

1. Does ISO 42030 already define the framework this project seeks?
2. Does it cover evaluation of changes to established architecture?
3. Can it be specialized into a lightweight Business Central review?
4. What remains genuinely domain-specific?

This note is based on publicly available official descriptions. It is not a substitute for the licensed standard.

# Source Status

ISO/IEC/IEEE 42030:2019 was published in July 2019. ISO states that it was reviewed and confirmed in 2025, remains current, and is scheduled to be revised.

Official sources:

- [ISO/IEC/IEEE 42030:2019 — ISO](https://www.iso.org/standard/73436.html)
- [IEEE/ISO/IEC 42030-2019 — IEEE Standards Association](https://standards.ieee.org/ieee/42030/7602/)

A new revision work item was approved in 2026.

# Executive Assessment

ISO 42030 is highly relevant and significantly narrows the originality claim available to this project.

The standard provides a generic framework for organizing and recording architecture evaluations.

Its official aims include:

- validating that architectures address stakeholder concerns;
- assessing architecture quality relative to intended purpose;
- assessing stakeholder value;
- determining whether architecture entities address intended purpose;
- providing architecture-related knowledge;
- assessing progress toward architecture objectives;
- clarifying the problem space and stakeholder needs;
- identifying risks and opportunities;
- supporting decision-making.

The standard explicitly states that specific evaluation frameworks can be derived from its generic framework.

## Central conclusion

> The project should probably not present “Behavior Change Review Framework” as a wholly new architecture-evaluation concept.

A stronger formulation is:

> The project proposes a lightweight Business Central-specific architecture evaluation framework, derived from general architecture evaluation principles and triggered by behavior-changing customizations.

## Remaining gap

The available official abstract does not specify:

- a Business Central trigger;
- effective execution flow terminology;
- `IsHandled`, interface or event-specific guidance;
- a minimal pull-request-level review;
- domain-specific concerns;
- practical code examples;
- proportionality criteria.

These remain legitimate areas of contribution.

# Scope of ISO 42030

The standard applies to architecture evaluation in enterprise, systems and software fields.

The entity evaluated can include:

- enterprise;
- organization;
- solution;
- system;
- subsystem;
- business;
- data;
- application;
- IT collection;
- mission;
- product;
- service;
- software item;
- hardware item;
- product line;
- family of systems;
- system of systems.

### Research implication

A Business Central customization, subsystem, extension ecosystem or posting capability can plausibly be scoped as an evaluation entity.

The project does not need to evaluate only complete solutions.

# Evaluation of Architecture, Not Merely Documentation

The official abstract emphasizes that ISO 42030 addresses evaluation of an architecture, not only the suitability of its architecture description.

Architecture-description suitability belongs primarily to conceptualization and elaboration under ISO 42020, though both evaluations may occur together.

### Research implication

The review must examine actual or intended system behavior, not only whether documentation is complete.

For Business Central this means:

- inspecting actual code and runtime paths;
- identifying effective event participation;
- understanding transaction behavior;
- comparing intended and actual extension semantics;
- examining cross-application effects.

A well-written impact record does not compensate for an unsound architecture.

# Evaluation Aims Mapped to the Project

## Validate that stakeholder concerns are addressed

Business Central concerns may include:

- accounting correctness;
- statutory localization;
- data integrity;
- security;
- operational diagnostics;
- effective extensibility;
- upgrade compatibility;
- maintainability;
- performance.

### Project use

A behavior-change review should explicitly map evidence to stakeholder concerns.

## Assess architecture quality relative to intended purpose

The same mechanism may be appropriate in one context and unacceptable in another.

For example:

- a tightly controlled customer-specific extension may accept a dependency that a reusable ISV app should not;
- a statutory localization may legitimately redefine behavior that a generic add-on should preserve;
- a temporary compatibility layer may accept technical debt that requires an expiry condition.

### Project use

The framework should assess fitness for purpose rather than enforce identical outcomes universally.

## Assess value to stakeholders

A replacement may improve one stakeholder's outcome while reducing value for another.

Example:

- faster implementation for one project;
- reduced composability for other extensions;
- higher maintenance burden for future upgrades.

### Project use

Trade-offs should be explicit.

## Determine whether the evaluated entity addresses its purpose

The review must ask whether the modified behavior still fulfils the business and technical purpose for which it exists.

### Project use

This prevents focusing only on implementation correctness.

## Provide architecture-related knowledge

A major practical problem in partner ecosystems is that behavior-changing decisions are hidden in subscribers and codeunits.

### Project use

A review record can make those decisions discoverable.

## Assess progress toward architecture objectives

Possible Business Central architecture objectives:

- reduce use of `IsHandled`;
- preserve extension composability;
- reduce copied base logic;
- improve upgrade safety;
- use interfaces for supported substitution;
- maintain transaction consistency.

### Project use

The framework could support organization-level improvement, not only one-time code review.

## Clarify problem space and stakeholder needs

Many poor extensibility choices originate from solving the wrong variation problem.

### Project use

Before choosing an alternative to `IsHandled`, the review should clarify whether the need is:

- augmentation;
- validation;
- substitution;
- orchestration;
- discovery;
- policy selection;
- post-processing;
- complete process replacement.

## Identify risks and opportunities

Risks may include:

- broken invariants;
- non-deterministic extension composition;
- unreachable events;
- behaviorally ineffective events;
- altered commit boundaries;
- missing telemetry;
- upgrade divergence.

Opportunities may include:

- introducing an interface;
- extracting a stable method;
- defining a business event;
- reducing override scope;
- creating reusable orchestration.

## Support decision-making

The framework should compare alternatives rather than simply detect violations.

# Architecture Evaluation as a Generic Framework

ISO states that the generic Architecture Evaluation framework can support the Architecture Evaluation process in ISO 42020.

It also states that specific frameworks may be derived from the generic framework.

### Central project implication

This gives a strong standards-aligned basis for creating:

> Business Central Behavior Change Architecture Evaluation Framework

The project can legitimately be presented as a **derived domain-specific framework**.

# Does ISO 42030 Already Contain the Proposed Framework?

## Conceptually, largely yes

It already establishes that architecture evaluations should:

- be organized and recorded;
- address stakeholder concerns;
- assess quality and purpose;
- identify risk;
- support decisions.

These are core ideas behind the proposed checklist.

## Operationally, no

It does not publicly provide Business Central-specific guidance for:

- detecting behavior change in AL code;
- tracing suppressed continuation;
- analyzing event semantics;
- evaluating effective extensibility;
- comparing `IsHandled`, interfaces, event types and workflows;
- determining review depth;
- creating practical examples from BaseApp and localization apps.

## Correct originality claim

The likely contribution is:

> domain specialization, trigger definition, operational guidance, proportionality and examples.

# The Trigger Question

The official public material describes what architecture evaluation is for, but does not state that every execution-flow modification automatically requires evaluation.

The project still needs to define a trigger.

## Candidate trigger

> A customization should enter behavior-change architecture evaluation when it may affect stakeholder concerns or architecture decisions beyond its local implementation boundary.

## Detection indicators

A review may be triggered when a change:

- suppresses established behavior;
- replaces an operation;
- delegates responsibility to another implementation;
- changes order of significant operations;
- introduces a new branch with materially different outcomes;
- changes transaction or failure semantics;
- changes supported extension participation;
- copies and diverges from upstream platform behavior;
- changes a published or relied-upon behavioral contract.

Execution-flow modification is therefore one indicator, not the complete definition.

# Proportionality

A critical challenge is scaling architecture evaluation.

## Full evaluation is not appropriate for every change

A single local condition change should not automatically require a large formal process.

## Candidate evaluation levels

### Level 0 — Local implementation review

No architecture-relevant concern is plausibly affected.

### Level 1 — Focused behavioral review

One bounded behavior and a small number of concerns are affected.

### Level 2 — Cross-extension architecture evaluation

Multiple applications, contracts or stakeholder groups are affected.

### Level 3 — Solution-level architecture evaluation

Core process, security, transaction architecture or long-term platform evolution is affected.

This level model is project-specific and requires validation.

# Effective Extensibility Under ISO 42030

The project's candidate term **Effective Extensibility** can be handled as an evaluation concern.

Possible question:

> Can an extension still participate meaningfully in the behavior that it was designed to influence after the customization is applied?

Evidence may include:

- technical event reachability;
- runtime path inclusion;
- use of subscriber-modified data;
- preservation of ordering assumptions;
- semantic correspondence of replacement events;
- dependency requirements;
- multi-extension interaction.

### Research implication

Effective Extensibility need not be promoted immediately to a new general architecture concept.

It can first be developed as:

> a Business Central-specific architecture evaluation concern.

This is a safer research position.

# Architectural Insignificance

ISO 42030 does not publicly provide a simple definition of an architecture-insignificant change.

The project must develop a practical threshold.

A change may remain local when:

- effects are contained in one implementation unit;
- no stakeholder concern outside the unit changes;
- no public or relied-upon contract changes;
- no cross-application behavior changes;
- transaction, security and upgrade assumptions remain stable.

This remains a candidate rule.

# Strongest Consequence for the Whitepaper

The whitepaper should not be framed as:

> Software architecture lacks a behavior-change evaluation framework.

It should be framed as:

> General architecture evaluation frameworks exist, but Business Central lacks a practical mechanism-independent specialization that connects everyday behavior-changing customizations to architecture evaluation.

This is a much more defensible thesis.

# Counterarguments

## Existing generic frameworks may already be sufficient

A critic may argue that competent architects can already apply ISO 42030, ATAM or similar methods.

Response:

The project must demonstrate why Business Central-specific operationalization adds value.

## Architecture evaluation may be too costly

A framework that is not proportional will be ignored.

## The trigger may be subjective

The framework needs observable indicators and examples.

## Effective extensibility may be one concern among many

The project should avoid elevating it into the sole objective.

## Partner teams may not use standards terminology

The final guideline may need a plain-language layer with standards mapping in the background.

# Findings

## Finding 1 — Accepted

A generic architecture evaluation framework already exists.

## Finding 2 — Accepted

Architecture evaluation explicitly concerns stakeholder concerns, purpose, quality, value, risks and decision support.

## Finding 3 — Accepted

Specific evaluation frameworks can be derived from ISO 42030.

## Finding 4 — Accepted

The project should present itself as a Business Central-specific specialization rather than a new general evaluation theory.

## Finding 5 — Candidate

Effective Extensibility can be introduced first as a Business Central evaluation concern.

## Finding 6 — Open

The exact detailed evaluation elements and required records remain unverified without licensed access.

## Finding 7 — Open

The review trigger and proportionality model remain the main possible contributions.
