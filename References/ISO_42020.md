# ISO_42020.md

# Purpose

This note evaluates **ISO/IEC/IEEE 42020:2019 — Software, systems and enterprise — Architecture processes** as a possible foundation for the Behavior Change Review Framework project.

The main questions are:

1. Does the standard already define the process that this project is attempting to introduce?
2. Does it define when architecture evaluation should occur?
3. Can its process descriptions be specialized for Business Central?
4. Which parts remain unavailable from the publicly accessible abstract and therefore require licensed access before stronger claims are made?

This note is based on publicly available official ISO and IEEE descriptions. It is not a substitute for the licensed standard.

# Source Status

ISO/IEC/IEEE 42020:2019 was published in July 2019. ISO states that it was reviewed and confirmed in 2025, remains current, and is now scheduled to be revised.

Official sources:

- [ISO/IEC/IEEE 42020:2019 — ISO](https://www.iso.org/standard/68982.html)
- [IEEE 42020-2019 — IEEE Standards Association](https://standards.ieee.org/ieee/42020/7601/)

# Executive Assessment

## What the standard clearly provides

According to the official abstract, ISO 42020 establishes process descriptions for:

- governance of architectures;
- management of a collection of architectures;
- architecting entities;
- enablement supporting the other architecture processes.

Official overview material also describes a coherent set of architecture processes including:

- governance;
- management;
- conceptualization;
- evaluation;
- elaboration.

The processes are intended to apply:

- to one project or multiple projects;
- throughout the life of an architecture;
- during any stage of the life cycle of the relevant architecture entities.

## What this means for the project

ISO 42020 substantially weakens any claim that software engineering lacks a general architecture-process framework.

The project should therefore not claim:

> Architecture-changing behavior modifications are currently outside established architecture-process standards.

A more defensible position is:

> ISO 42020 provides general architecture processes, but Business Central guidance does not appear to operationalize those processes for ordinary behavior-changing customizations introduced through extensibility mechanisms.

## What remains unresolved

From the publicly accessible abstract alone, it is not possible to establish:

- the detailed entry criteria for the Architecture Evaluation process;
- whether source-code behavior change is explicitly listed as a trigger;
- how architecture-significant changes are distinguished from local implementation changes;
- whether the standard prescribes mandatory concern categories;
- whether it defines a lightweight review suitable for pull requests.

These questions require direct access to the licensed text or an authorized detailed secondary source.

# Relationship to ISO 42010 and ISO 42030

The three standards form a useful conceptual chain:

```text
ISO 42010
Architecture description
        ↓
ISO 42020
Architecture processes
        ↓
ISO 42030
Architecture evaluation framework
```

This is not a strict lifecycle sequence, but it is a useful analytical relationship:

- ISO 42010 structures how an architecture can be described.
- ISO 42020 structures governance, management and architecting processes.
- ISO 42030 structures how architecture evaluations can be organized and recorded.

For this project, ISO 42020 is the first standard that directly addresses **process** rather than only description.

# Core Process Areas Relevant to the Project

## Architecture Governance

Architecture governance concerns the direction, oversight and control of architecture practice and architecture-related decisions.

### Relevance

A mechanism-independent Business Central guideline could become part of governance by defining:

- when a behavior-changing customization requires architecture attention;
- who is responsible for review;
- which evidence must be recorded;
- which deviations require explicit approval;
- how accepted risks are tracked.

### Limitation

The public abstract does not reveal whether ISO 42020 defines governance triggers at the granularity of code changes or extension mechanisms.

## Architecture Management

Architecture management concerns coordinated handling of architectures or collections of architectures over time.

### Relevance

Business Central partner solutions commonly consist of:

- Base Application;
- System Application;
- localizations;
- ISV applications;
- customer-specific extensions.

This resembles a collection of interacting architectures rather than one isolated codebase.

A behavior-changing customization may affect:

- one application;
- downstream applications;
- future upgrades;
- customer-specific behavior;
- the effective architecture of the full installation.

### Research implication

The project's ecosystem perspective is compatible with ISO 42020's stated applicability to collections of architectures and multiple projects.

## Architecture Conceptualization

Conceptualization is described in official overview material as one of the architecture processes.

### Relevance

Before selecting `IsHandled`, an interface, a workflow or another mechanism, a team may need to clarify:

- the problem to solve;
- stakeholders and concerns;
- alternatives;
- intended behavioral boundaries;
- which capabilities are variable;
- what must remain stable.

### Project implication

The proposed framework should not begin only after code has been written.

It may include an early conceptualization stage for planned behavior changes.

## Architecture Evaluation

Evaluation is explicitly part of the process set associated with ISO 42020.

### Relevance

This is the strongest overlap with the project.

The project's initial thesis was that a significant behavior change should trigger review. ISO 42020 already recognizes architecture evaluation as an architecture process.

### What remains potentially novel

The likely gap is not the existence of evaluation, but the operational connection:

```text
Business Central behavior-changing customization
                ↓
architecture evaluation process
```

The project may therefore define:

- the practical trigger;
- the lightweight scope;
- the Business Central concern set;
- the required evidence;
- the mapping to implementation mechanisms.

## Architecture Elaboration

Elaboration develops architecture decisions and descriptions in greater detail.

### Relevance

After a behavior change is accepted, the change may need elaboration through:

- execution-flow models;
- transaction models;
- event semantics;
- interfaces;
- extension-point contracts;
- upgrade strategy;
- implementation constraints.

### Project implication

The framework should distinguish:

- evaluating whether a change is acceptable;
- elaborating how the accepted change should be implemented and documented.

## Architecture Enablement

The official abstract states that the standard establishes an enablement process supporting other architecture processes.

### Relevance

For Business Central, enablement could include:

- coding guidelines;
- review templates;
- analyzers;
- example catalogs;
- training;
- architecture decision records;
- reusable viewpoints;
- partner education.

This directly supports the user's observation that isolated warnings about `IsHandled` are insufficient without practical operational support.

# Does ISO 42020 Define the Missing Review Trigger?

## Publicly supported conclusion

ISO 42020 defines architecture processes that include evaluation and apply throughout the life of architectures.

## Unsupported conclusion

The available public material does not prove that it explicitly states:

> Any change that alters an established effective execution flow must trigger architecture evaluation.

Therefore, this exact statement should not be attributed to ISO 42020.

## Most defensible interpretation

A significant behavior-changing customization can be treated as an input or trigger to an ISO 42020-aligned Architecture Evaluation process.

The identification of that trigger for Business Central remains a domain-specific specialization.

# Implications for the Project Hypothesis

Earlier hypothesis:

> A significant change to an established effective execution flow should trigger architectural review.

ISO-aligned refinement:

> A Business Central customization that may affect architecture-relevant stakeholder concerns should enter an appropriately scoped architecture evaluation process.

The reference to effective execution flow can remain as an important detection signal:

> Modification of an established effective execution flow is one practical indicator that architecture-relevant concerns may be affected.

This is safer than claiming that every flow change is automatically architectural.

# Candidate Business Central Process Specialization

A lightweight process aligned with ISO 42020 could contain:

## 1. Detect

Identify whether a customization:

- intercepts;
- suppresses;
- substitutes;
- delegates;
- reorders;
- or significantly augments established behavior.

## 2. Scope

Define:

- entity of interest;
- affected behavior;
- stakeholders;
- concerns;
- original and modified boundaries.

## 3. Conceptualize

Identify:

- desired outcome;
- alternatives;
- mechanism options;
- intended differences;
- constraints.

## 4. Evaluate

Assess:

- functional correctness;
- data invariants;
- transaction semantics;
- security;
- observability;
- extensibility;
- failure behavior;
- upgrade impact.

## 5. Decide

Record:

- accepted option;
- rationale;
- risks;
- rejected alternatives;
- unsupported capabilities.

## 6. Elaborate

Produce:

- implementation design;
- models;
- contracts;
- event or interface semantics;
- test strategy;
- documentation.

## 7. Govern

Define:

- reviewer;
- approval threshold;
- future review conditions;
- ownership;
- upgrade reassessment.

This is a project-specific interpretation, not text claimed to be required by ISO 42020.

# Important Finding: The Project Is Probably a Specialization

After reviewing the official scope of ISO 42020, the project's likely contribution should be framed as:

> A lightweight Business Central specialization of established architecture processes for behavior-changing customizations.

This is stronger than claiming a new general process.

It also provides a constructive relationship with existing standards:

- ISO supplies the process family;
- the project supplies the Business Central trigger, scope, concerns, examples and operational guidance.

# Counterarguments

## The standard may already contain more detail

Because the full text is not publicly available, the standard may already include process inputs, outputs or activities that cover much of the proposed framework.

This must be verified before publication.

## Applying architecture process to ordinary changes may be excessive

Not every behavior change warrants formal architecture governance.

The project still needs a proportionality model.

## Process terminology may be too heavy for partner teams

The framework must translate architecture-process concepts into lightweight development practices.

## Flow change may not be the correct primary trigger

A behavior can change architecturally without visible structural flow change.

The trigger may need to reference affected concerns rather than only execution structure.

# Findings

## Finding 1 — Accepted

General architecture governance, management, conceptualization, evaluation, elaboration and enablement processes already exist.

## Finding 2 — Accepted

The project should not claim to invent architecture evaluation as a process.

## Finding 3 — Candidate

The project can define a Business Central specialization that connects behavior-changing customizations to existing architecture processes.

## Finding 4 — Open

The exact architecture-evaluation entry criteria in ISO 42020 remain unverified from public material.

## Finding 5 — Accepted

Practical enablement—guidelines, examples, analyzers and review templates—is a legitimate part of the problem.
